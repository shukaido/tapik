1-2b. 동적 적립률 3모드 (v5 신규)

v5 핵심 변경: 고정 적립률(solo 5% / joiner 6% / creator 8%) → Room 인원 수(N)에 따라 동적으로 계산되는 3가지 모드. Admin에서 실시간 전환 가능.
모드이름설명Joiner 적립률Creator 적립률
default 디폴트 3단계 티어 — 상시 적용 N≤4: 5.50% · N≤6: 5.75% · N≥7: 6.00% N≤4: 6.00% · N≤6: 6.50% · N≥7: 7.00%
spot 스팟 6단계 리니어 — 이벤트 기간 한정 5.00% + (N-3)×0.20% (3명 5.0% → 8명 6.0%) 5.50% + (N-3)×0.40% (3명 5.5% → 8명 7.5%)
hotspot 핫스팟 Creator 보너스 극대화 — 바이럴 부스트 target_net 8.5%→10% 리니어 보간 Joiner × 2 (최대 10%) 
// v5 적립률 계산 코드 (Consumer + Admin 동기화)
const REWARD_MODES = {
  default: {
    name: '디폴트', desc: '3단계 티어 — 상시 적용',
    getRate(n) {
      if (n <= 4) return { joiner: 5.50, creator: 6.00 };
      if (n <= 6) return { joiner: 5.75, creator: 6.50 };
      return { joiner: 6.00, creator: 7.00 };
    }
  },
  spot: {
    name: '스팟', desc: '6단계 리니어 — 이벤트 기간',
    getRate(n) {
      const j = 5.0 + Math.max(0, n - 3) * 0.2;
      const c = 5.5 + Math.max(0, n - 3) * 0.4;
      return { joiner: +j.toFixed(2), creator: +c.toFixed(2) };
    }
  },
  hotspot: {
    name: '핫스팟', desc: 'Creator 보너스 극대화',
    getRate(n) {
      const target_net = 8.5 + (n - 3) * ((10 - 8.5) / 5);
      const j = target_net / (1 + 1); // 단순화
      const c = Math.min(j * 2, 10);
      return { joiner: +j.toFixed(2), creator: +c.toFixed(2) };
    }
  }
};

1-3. 설계 원칙

① 폐쇄 루프: 모든 주문·결제·정산이 Tapik 내부에서 완결. 외부 POS·배달앱 의존 제로.

② 실시간 우선: Firestore onSnapshot 기반. 주문 상태·Room 멤버·적립률·대기열 — 모두 실시간 구독. 폴링 제로.

③ Room 퍼스트: Room은 선택 기능이 아닌 핵심 바이럴 엔진. 모든 주문 플로우에 Room 진입점 존재. 최소 3명.

④ 태블릿 네이티브: 머천트 앱은 Lenovo Tab M7 (7인치, 1024×600) 전용 설계. 터치 타겟 최소 48px.

⑤ 오프라인 그레이스: Firestore 오프라인 캐시 + 복구 시 자동 동기화.

⑥ 동적 인센티브: 적립률을 고정하지 않고 Admin이 실시간 전환 가능한 3모드 체계. 시장 반응에 따라 즉시 조정.

2-1. 주문 상태 머신 (Order State Machine)

┌─────────────────────────────────────────────┐ │ ORDER STATE MACHINE │ └─────────────────────────────────────────────┘ [CREATED] ──consumer.pay()──▶ [PAID] ──consumer.placeOrder()──▶ [ORDERED] │ │ │ consumer.cancel() │ merchant.accept() ▼ ▼ [CANCELLED] [PREPARING] (환불 처리) │ │ merchant.markReady() ▼ [READY] │ │ merchant.scanQR() │ OR consumer.showQR() ▼ [PICKED_UP] │ │ auto: points credited ▼ [COMPLETED] 타임아웃 규칙: • PAID → 24시간 내 ORDERED 안 하면 → 자동 CANCELLED + 전액 환불 • ORDERED → 5분 내 merchant.accept() 없으면 → 알림 재전송 + 7분 후 자동 CANCELLED • READY → 60분 내 PICKED_UP 안 하면 → NO_SHOW 처리 (환불 없음, 포인트 미적립)

2-2. Room 상태 머신 (v5 — 6단계)

v5 핵심 변경: v4의 4단계(WAITING→LOCKED→ACTIVE→CLOSED)를 6단계로 세분화. ROOM_MIN 2→3. paidMembers 추적. Creator 방폭 정책 추가.
┌──────────────────────────────────────────────────────────┐ │ ROOM STATE MACHINE (v5 — 6단계) │ └──────────────────────────────────────────────────────────┘ creator.createRoom() │ ▼ [WAITING] ───── min 3명 참여 ────▶ [READY] ──── Creator activateOrdering() ────▶ [ORDERING] │ │ │ │ creator.cancel() │ (Joiner: 대기만) │ 전원 markMemberPaid() │ (waiting에서만 가능) │ ▼ ▼ │ [ALL_PAID] [DELETED] │ │ │ │ auto: 전원 PICKED_UP │ ▼ │ [CLOSING] │ │ │ │ auto 또는 수동 │ ▼ │ [DELETED] │ └── 모든 Joiner 이탈 시 ──▶ [CLOSING] ──▶ [DELETED] ⚠️ Creator 방폭: Creator는 waiting 상태에서만 Room 취소(삭제) 가능. ready 이후에는 Creator가 Room을 떠날 수 없음. 모든 Joiner가 먼저 떠나야 Room이 closing → deleted 전환. 정산 계산 로직 (매주 월요일):
  grossGMV       = sum(orders[Mon-Sun].total)
  commission     = grossGMV × MERCHANT_COMM
  pointsSubsidy  = 0  (포인트는 Tapik 부담, 매장 공제 아님)
  refunds        = sum(cancelled_orders.total)
  netSettlement  = grossGMV - commission - refunds
  → 매장 계좌로 이체

6 Room 프로토콜 — v5 전체 재설계

Room 2종 (v5):
🌐 Public Room — 누구나 생성 · 누구나 참여 · 위치 기반 탐색
🔒 Private Room — 비밀번호 입장 · 연락처 퍼미션 불필요
Firestore rooms 컬렉션에 실시간 저장. password 필드 존재 여부로 Public/Private 구분.
v5 Room 핵심 변경:
① 상태 머신 6단계: waiting → ready → ordering → all_paid → closing → deleted
② ROOM_MIN = 3: 최소 인원 2→3 상향 (콜드스타트 임계점)
③ paidMembers: 결제 완료 멤버 배열로 개별 추적 + 체크마크 UI
④ Creator 방폭: Creator는 waiting에서만 취소 가능. ready 이후 이탈 불가.
⑤ 자동 전환: joinRoom → 3명 충족 시 자동 waiting→ready. 전원 결제 시 자동 ordering→all_paid.
⑥ 동적 적립률: Room 생성 시 활성 모드 스냅샷 저장 → 해당 모드 + 인원 수로 적립률 계산.

2. 설정 Room 설정 화면:
• 최대 인원 (3-8)
• 🔒 Private: 비밀번호 4자리 입력
• 메시지 입력 (선택)		 로컬 state. Private: createPassword state. — —
3. Room 생성 "Room 열기" 확인 버튼 Firestore Room 문서 생성 
addDoc(collection(db, 'rooms'), {
  storeId, creatorId: uid,

4. 공유 Room 대기 화면: 참여자 리스트 + 상태 뱃지

🔒 Private: "비밀번호: {XXXX}를 친구에게 공유해"		 Firestore onSnapshot으로 실시간 멤버 업데이트 — Firestore 실시간 → 모든 Consumer에 Room 리스트 업데이트
5. 대기 (waiting) 참여자 실시간 업데이트. "최소 3명 필요" onSnapshot on Room 문서 멤버 ≥ 3 시 자동 status → 'ready' 전 멤버에게 상태 변경 실시간 반영

6-2. Room 참여 플로우 (Joiner)

단계화면클라이언트Firestore
1. 진입 🌐 Public: Room 탭 리스트에서 선택
🔒 Private: "비밀번호 입장" → 비밀번호 입력 모달		 Room 카드 탭 또는 비밀번호 검증 Room 문서 조회 → 🔒이면 password 대조
2. 참여 "참여하기" 버튼 joinRoom() 함수 호출 
// members 배열에 수동 추가 (arrayUnion 대신)
const snap = await getDoc(roomRef);
const members = snap.data().members || [];
if (!members.includes(uid)) {
  members.push(uid);
  await updateDoc(roomRef, { members });
}
// 자동 전환: members.length >= ROOM_MIN → status: 'ready'
3. 주문 (ordering 상태) Creator가 활성화하면 "주문하기" 버튼 활성 각자 메뉴 선택 → 결제 결제 완료 시 markMemberPaid(uid) → paidMembers에 추가
4. 완료 전원 결제 완료 → all_paid → 픽업 대기 주문 상태 화면으로 전환 전원 PICKED_UP → status: 'closing'

6-3. Room 상태 전환 & 트리거

이벤트조건Firestore 처리전체 멤버 영향
waiting → ready members.length ≥ ROOM_MIN (3) 자동 전환: joinRoom() 내에서 체크 → updateDoc({status: 'ready'}) "Room 준비 완료! Creator가 주문 시작을 기다려" + 상태 뱃지 변경
ready → ordering Creator가 "주문 시작" 버튼 탭 activateOrdering() → updateDoc({status: 'ordering'}) 전원에게 "주문 시작! 각자 주문을 넣어" + 주문 버튼 활성
ordering → all_paid paidMembers.length === members.length markMemberPaid() 내에서 체크 → updateDoc({status: 'all_paid'}) 전원에게 "전원 결제 완료! ✓" + 주문 처리 대기
all_paid → closing 전원 PICKED_UP 완료 updateDoc({status: 'closing'}) "Room 완료! 포인트 적립 중..."
closing → deleted Creator "Room 종료" 또는 자동 destroyRoom() → updateDoc({status: 'deleted'}) Room 아카이브. 히스토리에서 확인 가능.

6-4. Room 엣지케이스 (v5)

케이스처리
Creator 방폭 v5 신규 Creator는 waiting 상태에서만 Room 취소(삭제) 가능. ready 이후에는 이탈 불가. "방장은 나갈 수 없어!" 토스트. 모든 Joiner가 먼저 떠나야 Creator가 Room 종료 가능.
Joiner 이탈 (자유) leaveRoom() → members에서 제거. 이탈 후 members.length < ROOM_MIN이면 → 자동 ready→waiting 다운그레이드 (구현에 따라 유지 또는 다운그레이드). 모든 Joiner 이탈 → status: 'closing'.
PICKED_UP 후 자동 이탈 주문 픽업 완료 시 자동으로 Room에서 이탈. 마지막 멤버 이탈 시 closing → deleted.
30분 타임아웃 (waiting) 최소 인원 미달 시 Room → deleted. 참여자 각자 solo 전환.
비밀번호 불일치 (Private Room) "비밀번호가 틀려!" 토스트. 참여 거부. 재입력 가능.
한 멤버 NO_SHOW 해당 멤버만 NO_SHOW 처리. 다른 멤버 포인트율은 유지 (Room 인원 수 기반).

14 Merchant 데이터 범위 확정

14-1. Tapik 채널 전용

확정: Merchant 태블릿에는 Tapik 채널 매출만 표시. Walk-in 현금, DuitNow, 배달앱 등 비-Tapik 매출은 표시하지 않는다. POS가 없으므로 외부 데이터를 받을 수 없고, 수동 입력도 제공하지 않는다.

이유:

① POS 제공사가 데이터를 안 준다 — 피봇 사유 그 자체.

② 수동 입력은 매장 사장님에게 부담 → 태블릿 앱의 "쉽고 빠름" 가치 훼손.

③ Tapik 채널만 정확하게 보여주는 게 데이터 신뢰도 확보에 유리.

매장 사장님 관점: "오늘 Tapik으로 얼마 벌었나?" = 태블릿에서 확인. "오늘 전체 매출?" = 기존 방식대로 (현금 세기, 은행 앱 확인). Tapik이 성장하면 전체 매출 중 Tapik 비중이 자연스럽게 올라가므로, 장기적으로 태블릿이 사실상의 매출 대시보드가 된다.

이 문서의 범위: Tapik v5 앱 전체 — Consumer 5탭, Merchant Tablet 6탭, Admin 7탭의 모든 화면, 모든 버튼, 모든 인터랙션에 대한 클라이언트→Firestore→크로스앱 로직을 정의했다.

v5 핵심: Firebase Auth + Firestore 서버리스 아키텍처, Room 6단계 상태 머신(waiting→ready→ordering→all_paid→closing→deleted), 동적 적립률 3모드(Default/Spot/Hotspot), paidMembers 추적, Creator 방폭, Private Room 비밀번호, TP Cap 100,000, 실시간 매장 대기열, 적응형 "나의 현황" 홈 섹션.

Room은 Public/Private 2탭, Private은 비밀번호 입장. 위치 퍼미션 확정, 연락처 퍼미션 완화. 첫 실행 온보딩 7단계(Google 로그인), i18n 아키텍처, 면책 조항까지 포함. API 엔드포인트 60개+ (Firestore 직접 연동으로 REST 엔드포인트 감소), 이상탐지 규칙 8개.

빌드 완료: Consumer v5 · Merchant v5 · Admin v5 — 3개 앱 Firebase 연동 완료. BroadcastChannel 크로스앱 동기화 확인.
